# v5.0.1 - Follow-up checklist

Items deferred from the v5.0 ship. None are blockers - the multi-agent audit on the v5.0 branch found no critical or important issues that required holding the release. This list captures the "would-be-nice" gap closures, ordered by impact + reasonable effort.

The label "v5.0.1" is aspirational. Most items realistically land in v5.1+ when there's a natural reason to bundle them; a few are quick enough to fit in a patch release if a real v5.0 regression forces one.

## Test coverage gaps

The audit identified production paths without unit tests. Total suite is 321 tests passing today; everything below would extend that.

### LegacyHelperCleanupService
- **File:** `Shared/Services/LegacyHelperCleanupService.swift`
- **Status:** Manual E2E verified pre-ship (PR #144 task #33). No unit tests.
- **Why it matters:** One-shot v4.x -> v5.0 migration. If broken, upgrading users see onboarding again or have leftover LaunchAgent cruft.
- **What's needed:** DI refactor (inject `FileManager` + `UserDefaults` + `Process` abstractions), then test:
  - `migratePrefsIfNeeded()`: read sandbox plist -> write to real UserDefaults -> set migrated flag (run-once semantics)
  - `runIfNeeded()`: detect legacy artifacts -> spawn `launchctl bootout` -> remove plist + `keychain-token.json`
  - Both gated by UserDefaults flag, both no-op on fresh installs

### SharedFileService
- **File:** `Shared/Services/SharedFileService.swift`
- **Status:** No unit tests.
- **Why it matters:** Critical cross-process IO between app + widget; carries the whole shared usage cache + theme + settings mirror.
- **What's needed:**
  - `containerURL` nil -> home-relative fallback (Personal Team / pre-cert path)
  - Migration `com.claudeusagewidget.shared` -> `com.tokeneater.shared` -> App Group
  - Cache invalidation
  - `NSFileCoordinator` read/write safety under concurrent app + widget access

### SessionHistoryService
- **File:** `Shared/Services/SessionHistoryService.swift`
- **Status:** No unit tests (the JSONL parsing path is integration-heavy: real FS, `enumerateLines`, mtime cache).
- **What's needed:**
  - mtime cache hit (file.mtime == cached.mtime -> reuse) and miss (re-parse)
  - Date bucketing (hourly for 24h range, daily for 7d/30d/90d)
  - `input_tokens` line pre-filter
  - `cwd` extraction vs `decodedProject` fallback (recently fixed - the static helper is now legacy-only)

### HistoryStore + MonitoringInsightsStore
- **Files:** `Shared/Stores/HistoryStore.swift`, `Shared/Stores/MonitoringInsightsStore.swift`
- **Status:** No unit tests. Lower risk - both are thin drivers around `SessionHistoryService`.
- **What's needed:** range/filter change reload, `isLoading` / `hasLoadedOnce` transitions, `applyFilter` model-family scaling, `warmIfStale` 60s cache.

## Code cleanup

### Drop the legacy migration once the v5.0 population has upgraded
- Wait ~2-4 weeks after v5.0 ship + monitor adoption.
- Delete `Shared/Services/LegacyHelperCleanupService.swift` entirely.
- Remove the `runIfNeeded()` + `migratePrefsIfNeeded()` calls from `RaiUsageApp.init()` / `AppDelegate`.
- Drop the `legacyHelperCleanupDone` + `legacySandboxPrefsMigrated` UserDefaults flags from any stored-prefs survey.
- Remove the Personal Team workaround section from `CLAUDE.md`.
- The `AppGroup` / fallback split in `SharedFileService` can stay - it correctly handles dev builds.

### CommonCrypto -> CryptoKit migration
- **File:** `Shared/Services/ElectronDecryptionService.swift`
- `import CommonCrypto` is a "private Apple framework" - works on macOS 14+ but isn't on the official public SDK list.
- macOS 14+ supports CryptoKit's AES-128-CBC + PBKDF2 equivalents.
- Future-proofing only - no immediate functional risk.

### `decodedProject` legacy fallback
- **File:** `Shared/Services/SessionHistoryService.swift` (the static helper)
- Now only used when an older JSONL line lacks `cwd`. Confidence is high that all current Claude Code installs write `cwd`.
- Once we've shipped a couple releases without seeing the fallback log fire, drop the static helper entirely.

### Re-evaluate `CredentialsFileReader`
- **File:** `Shared/Services/CredentialsFileReader.swift`
- Still wired into `TokenProvider`'s fallback chain (covered by `TokenProviderTests`).
- Targets Linux / Windows / very old macOS installs without `~/.claude/projects/`.
- Confirm it's still load-bearing or strip it - one less code path to maintain if not.

### `@Observable` migration (post-v5.0 stable, ~v5.2-v5.3)
- **Scope:** every store + every consuming view (~50+ files mechanical refactor).
- **Why we're on `ObservableObject` today:** `@Observable` triggered a 100% CPU freeze (infinite re-evaluation loop) in Release builds compiled with Swift 6.1.x (Xcode 16.4, current CI toolchain). Documented in `CLAUDE.md` under "@Observable interdit". The bug doesn't repro in Debug or with Swift 6.2+ (Xcode 26+).
- **Why migrate eventually:**
  - Per-property granularity vs per-object `objectWillChange` invalidation. SwiftUI re-evaluates only the views that actually depend on the changed property.
  - Cleaner code: `@Observable class Store { var foo: Int }` instead of `class Store: ObservableObject { @Published var foo: Int }`. Drop `@StateObject` / `@ObservedObject` / `@EnvironmentObject` in favour of `@State` / `@Bindable` / `@Environment(Store.self)`.
  - Apple-recommended modern API since iOS 17 / macOS 14.
- **Prerequisites before refactor:**
  - GitHub Actions runner with Xcode 26+ (likely `macos-26` image once available, otherwise self-hosted).
  - Confirm the Swift 6.1 freeze is gone: pick one isolated store (e.g. `HistoryStore`), migrate it to `@Observable` locally with Xcode 26, build Release, run the app for 30-60 min, watch the CPU. If stable, extend store by store.
  - `MACOSX_DEPLOYMENT_TARGET` stays at 14.0 (`@Observable` available since macOS 14, no compat regression for end users).
- **Mechanical conversion per store:**
  - `class XStore: ObservableObject { @Published var foo: ... }` -> `@Observable class XStore { var foo: ... }`
  - `@StateObject private var store = XStore()` -> `@State private var store = XStore()` at the App / root level
  - `@EnvironmentObject var store: XStore` -> `@Environment(XStore.self) var store`
  - `.environmentObject(store)` -> `.environment(store)`
  - `@ObservedObject var store: XStore` -> just `var store: XStore`, or `@Bindable var store: XStore` when bindings are needed
  - Bindings `$store.foo` keep working via `@Bindable`.
- **Risk:** medium - mechanical but touches ~50+ files; manual smoke test mandatory across every space + popover + widget. The reward is real but the reintroduction risk for the freeze bug is the dominant concern. Don't bundle this with anything else.

## UX polish

### Range pill bottom hit-zone (History view)
- **File:** `RaiUsageApp/Windows/History/HistoryView.swift` (`rangePicker`)
- **Symptom:** clicking just below the text on inactive range pills (24h / 7d / 30d / 90d) doesn't trigger the action. Top edge works.
- **Investigated:** added `.contentShape(Rectangle())`, then made the background always-present (clear when inactive). Neither closed the dead zone.
- **Hypothesis:** SwiftUI macOS plain-button hit-testing artifact, possibly tied to how the `.padding(3)` wrapper renders the rounded rectangle. Top vs bottom asymmetry is the puzzle - if it were straight padding inheritance both edges would behave identically.
- **Things to try:** bump vertical padding from 5 -> 8pt and remove wrapper outer padding entirely so each pill is flush with the wrapper border (the active highlight would then need a slight inset to keep visual breathing room). Or use a custom `Button` with `onTapGesture` on the label and skip `.buttonStyle(.plain)`.

### `CredentialsFileReader` UX (if kept)
- If the reader stays, surface a clearer hint when it falls through to it ("reading credentials.json fallback") so support can diagnose Linux/Windows install issues quickly.

## Hardening (lower priority)

### Concurrency-checked sendability
- `@unchecked Sendable` is currently used on `TokenProvider`, `SecurityCLIReader`, `SharedFileService`, `ElectronDecryptionService`.
- Mutable state inside (cached token, derived key, in-memory cache) is accessed from async contexts.
- Today this is safe because all callers go through `@MainActor` stores. The conformance hides that invariant from the compiler.
- Future: replace with proper `actor` isolation or `nonisolated(unsafe)` + an explicit lock.

### Widget `NSFilePresenter` registration
- The widget reads via `NSFileCoordinator` but doesn't register as `NSFilePresenter`.
- This means it gets the lock during a coordinated read but isn't notified of changes proactively.
- Acceptable today because WidgetKit's timeline model is pull-only - the widget refreshes on its own schedule, not on file change.
- Worth documenting in `SharedFileService` so a future contributor doesn't expect push behaviour.

### Electron derived-key file race
- **File:** `Shared/Services/ElectronDecryptionService.swift` (around line 261)
- `setAttributes([.posixPermissions: 0o600], ...)` runs after `write(to:)` returns - tiny window where the file is world-readable.
- `.completeFileProtection` already requires the device to be unlocked, so practical risk is minimal.
- Stronger fix: write inside a locked critical section, or move the derived key out of disk entirely (Keychain item with the right access group).

## Out of scope (intentionally)

- **`MARKETING_VERSION` management:** manual bump in `project.yml` before each tag stays. The `release-swift` skill automates the bump + bundling.
- **Embedded Python in `ProcessResolver`:** the tmux pane resolver embeds a tiny Python script as a string. Works fine; just looks unusual in code search. No action needed.
- **Sparkle adoption:** `UpdateService` rolls a custom appcast pipeline that's been stable across many releases. Adopting Sparkle would unlock delta updates + phased rollouts but is a meaningful refactor for a small win.
